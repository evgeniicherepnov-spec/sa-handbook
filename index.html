Раздел 0 · Старт
Кто такой системный аналитик?
Понимание роли — первый шаг. Разберёмся, чем занимается СА и чем отличается от смежных ролей.
Одна фраза о профессии

Системный аналитик — это инженер-переводчик между бизнесом и разработкой. Он принимает бизнес-боль, превращает её в чёткие системные требования, API-контракты и проверяет готовую систему.

Бизнес-аналитик (BA)
Уровень: Бизнес
  • Зачем делаем? Считает ROI и ценность
  • Общается с C-level и стейкхолдерами
  • Описывает бизнес-процессы AS-IS / TO-BE
  • Формирует бизнес-требования (BRD)
  • Не лезет в код и базы данных
Системный аналитик (SA)
Уровень: Система
  • Как делаем? Схемы данных, брокеры, API
  • Общается с разработчиками и QA
  • Проектирует архитектуру на уровне сервисов
  • Пишет ТЗ, Swagger-контракты, UML-схемы
  • Читает JSON, SQL, логи, понимает код

Раздел 0.5 · Основы
SDLC — Жизненный цикл разработки ПО
Software Development Life Cycle — это структурированный процесс создания программного обеспечения от идеи до вывода из эксплуатации. СА участвует на каждом этапе, но с разной интенсивностью.
Зачем СА знать SDLC

SDLC — это общий язык с командой. Когда PM говорит "мы на этапе design" или разработчик спрашивает "это уже в implementation?" — СА должен понимать где он находится, что от него ожидают и какие артефакты нужно подготовить прямо сейчас.

SDLC в Waterfall vs Agile

Фазы SDLC одинаковые в обоих подходах — но организованы принципиально по-разному.

Waterfall
  • Фазы выполняются последовательно и однократно
  • Все требования собираются до начала разработки
  • СА создаёт полное ТЗ в самом начале
  • Изменения требований — дорого и болезненно
  • Результат виден только в конце проекта
  • Подходит для: строительство, госзакупки, hardware
Agile (Scrum)
  • Все фазы повторяются в каждом спринте (1-2 недели)
  • Требования собираются итеративно
  • СА работает с бэклогом, уточняет требования постоянно
  • Изменения — норма, встроены в процесс
  • Рабочий продукт виден после каждого спринта
  • Подходит для: продуктовые компании, стартапы, SaaS
Как это выглядит на практике

Waterfall: СА 3 месяца собирает требования → пишет 80-страничное ТЗ → передаёт разработчикам → появляется через полгода на приёмке.

Agile: СА каждый спринт уточняет 5-10 User Stories → присутствует на grooming и planning → обновляет документацию по ходу → участвует в review каждые 2 недели.

Методологии разработки
СА работает по-разному в зависимости от того, как команда управляет проектом.
Waterfall (Водопад)
  • Строгие этапы: Сбор → Дизайн → Код → Тест.
  • СА пишет огромное ТЗ до старта кода.
  • Изменения вносить долго и дорого.
Госзаказ, Банки (legacy)
Scrum (Agile)
  • Спринты по 1-4 недели.
  • СА пишет требования порциями (User Story).
  • Ежедневные Daily митинги.
Продуктовые IT-компании
Kanban
  • Нет спринтов, непрерывный поток на доске.
  • WIP-лимиты (нельзя брать > X задач).
  • Задачи делаются по мере поступления.
Поддержка, DevOps

Важные термины Scrum

Бэклог и Декомпозиция
▼

Epic (Эпик): Огромная задача ("Система лояльности").

Feature (Фича): Кусок эпика ("Начисление баллов").

User Story: Сценарий на 1 спринт ("Как юзер, хочу видеть баллы...").

Task: Техническая задача для кодера ("Написать GET-метод API").

Story Points (Оценка в попугаях)
▼

Оценка сложности и рисков, а не часов. Числа Фибоначчи: 1, 2, 3, 5, 8, 13. Если задача оценена больше 8 SP — аналитик обязан её разбить на две (декомпозировать).

SAFe — Scrum для больших компаний
▼

Scaled Agile Framework (SAFe) — фреймворк масштабирования Agile на уровень всей организации. Используется когда над одним продуктом работают 50–1000+ человек в нескольких командах. Стандарт в банках, телекоме и крупном enterprise 2024.

Уровни SAFe:

  • Team Level — обычные Scrum-команды (2-недельные спринты)
  • Program Level — Agile Release Train (ART): 5–12 команд синхронизируются в Program Increment (PI) на 8–12 недель
  • Portfolio Level — стратегические инициативы, бюджеты, эпики масштаба компании

Роль СА в SAFe: System Architect / Business Analyst на уровне программы. Пишет Feature-требования (не User Story — они на уровне команды), участвует в PI Planning (квартальное планирование всех команд).

Если работаешь в крупной компании — скорее всего используется SAFe или его упрощённый вариант. Знание этого фреймворка отличает мидла от джуна в enterprise.

Hybrid-подходы — реальность большинства компаний
▼

Чистый Scrum или чистый Waterfall на практике встречаются редко. Большинство компаний используют гибриды: Waterfall для планирования и контрактов + Agile для разработки. СА должен уметь работать в любой среде и переключаться между формальной документацией (для ТЗ) и User Story (для беклога).

Типичные гибриды:
  • ScrumBan — Scrum-спринты + Kanban-доска для потока задач
  • Water-Scrum-Fall — Waterfall планирование → Scrum разработка → Waterfall сдача
  • SAFe + Scrum — PI-планирование на квартал + спринты внутри
Раздел 2 · Стейкхолдеры
Стейкхолдеры — полный разбор
Стейкхолдер — любой человек или группа, кто влияет на проект или испытывает его влияние. Главный навык СА — правильно выявить всех и выстроить работу с каждым.

Кто такие стейкхолдеры — полная карта

💰
Спонсор / Заказчик
Финансирует проект. Говорит «зачем». Главный критерий — ROI и бизнес-результат. С ним согласовывают цели и бюджет.
Пример: CMO, CEO, Директор по продукту
📦
Product Owner (PO)
Владелец бэклога. Приоритизирует задачи. Принимает итоговые решения по требованиям. СА работает с ним ежедневно.
Связь: несколько раз в день
👤
Конечные пользователи
Будут работать с системой. Хотят удобство и скорость. Часто молчат о проблемах — нужно наблюдение и интервью.
Пример: операторы, менеджеры, покупатели
⚙️
Команда разработки
Backend, Frontend, QA, DevOps. Получают требования от СА. Хотят чёткость, конкретику и отсутствие противоречий.
Связь: ежедневно на стендапе

Раздел 3 · Требования
Виды требований (Иерархия)
Требования — это то, ЧТО должна делать система и КАК ХОРОШО она должна это делать.
BR Бизнес-требования (Зачем?)
▼

Высокоуровневая цель бизнеса. Пример: «Увеличить конверсию в покупку на 15% за счет внедрения покупки в 1 клик».

BRules Бизнес-правила
▼

Политики, формулы и законы, которые определяют работу бизнеса независимо от того, существует ли IT-система вообще. Система должна им строго подчиняться.

  • Правило расчета: Скидка VIP-клиента составляет 15%, но не более 5000 рублей.
  • Ограничение: Пользователь младше 18 лет не может купить алкоголь (согласно закону РФ).
  • Событие: Если платеж просрочен на 3 дня, начисляется пени 1% за каждый день.

Аналитик собирает их в отдельный реестр (матрицу бизнес-правил), чтобы кодеры могли вынести эту логику в отдельные микросервисы (Decision Engines).

UR Пользовательские требования
▼

Что юзер сможет делать в системе. Пример: «Как клиент, я хочу привязать карту для быстрой оплаты».

FR Функциональные (Что делает система?)
▼

Детальное поведение системы.

  • Система должна валидировать номер карты по алгоритму Луна.
  • Система должна списать сумму со счета по API банка.
NFR Нефункциональные (Атрибуты качества)
▼
  • Производительность: Система держит 1000 запросов в сек (RPS).
  • Доступность: SLA не менее 99.9% времени.
  • Безопасность: Пароли хранятся в хэшированном виде (Bcrypt).
SR Системные требования
▼

Требования к среде исполнения (железо, ОС, браузеры). Чаще всего пишутся для коробочного (On-Premise) ПО, которое устанавливается на сервера заказчика.

  • Система должна разворачиваться на базе ОС Ubuntu 22.04 LTS.
  • СУБД должна быть PostgreSQL версии не ниже 14.0.
  • Минимальные аппаратные требования: 8 CPU, 32 GB RAM, 500 GB SSD.

8 атрибутов качества требования — по Карлу Вигерсу

Хорошее требование — не просто текст. Вигерс выделяет 8 атрибутов, которые должны выполняться для каждого требования. Если хотя бы один нарушен — требование вернут на доработку или оно вызовет баг на проде.

Complete
1. Полное
Описывает всё необходимое поведение — включая исключения и граничные случаи.
❌ «Система должна валидировать данные»
✅ «Система проверяет email по RFC 5322 и показывает ошибку, если формат неверный»
Correct
2. Корректное
Точно описывает реальную потребность стейкхолдера. Проверяется только с тем, кто это требование поставил.
Частая проблема: СА написал то, что понял, а не то, что имелось в виду. Решение — валидация с заказчиком на ревью.
Feasible
3. Реализуемое
Технически и финансово достижимо в рамках бюджета и стека проекта.
❌ «Система должна обрабатывать запросы за 0 мс»
Если сомневаешься — согласуй с Tech Lead ещё до записи в ТЗ.
Necessary
4. Необходимое
Удаление приводит к потере ценности для бизнеса или пользователя. Нет необходимости — нет требования.
Спроси: «Что произойдёт, если этого не будет?» Если ответ «ничего» — требование лишнее.
Prioritized
5. Приоритизированное
Каждому требованию присвоен приоритет. Без этого разработчики сами решают, что делать первым.
Минимум: Must / Should / Could по MoSCoW. Идеально: числовой приоритет + обоснование бизнес-ценности.
Unambiguous
6. Однозначное
Только одна возможная интерпретация. Стоп-слова: «быстро», «удобно», «при необходимости», «как правило».
❌ «Система должна быстро отвечать»
✅ «Время ответа API ≤ 200 мс при нагрузке до 500 RPS»
Verifiable
7. Проверяемое
Можно написать тест, который однозначно подтвердит выполнение. Непроверяемое требование — это пожелание, не требование.
Правило: если QA не может написать тест-кейс — требование нужно переписать.
Consistent
8. Непротиворечивое
Не конфликтует с другими требованиями в спецификации. Противоречия всплывают на code review или на проде.
❌ FR-01: «Сессия живёт 30 мин» + NFR-05: «Пользователь не перелогинивается в течение рабочего дня»
Стоп-слова в требованиях — красные флаги по Вигерсу

Если видишь эти слова — требование нужно переформулировать: «быстро», «удобно», «интуитивно», «при необходимости», «как правило», «соответствующий», «и/или», «поддерживать», «гибко», «легко», «достаточно», «максимально». Каждое из них — источник споров на приёмке.

Раздел 3.1 · Документация
Структура Спецификации (SRS)
Software Requirements Specification — главный документ аналитика (по ГОСТ 34, IEEE 830 или шаблонам в Confluence). Ниже представлена идеальная современная структура документации на фичу.
Главное правило хорошего ТЗ

Спецификация должна быть однозначной, полной, проверяемой и непротиворечивой. Если что-то не описано в ТЗ — разработчик сделает это на своё усмотрение (и почти всегда не так, как нужно бизнесу).

1. Бизнес-контекст и Цели
▼

Проблема: Какую боль решаем? (Текущий ручной ввод занимает 20 минут).

Цель: Снизить время до 2 минут.

Глоссарий: Расшифровка терминов (чтобы "Пользователь" и "Клиент" не путались).

2. Бизнес-процессы (AS-IS / TO-BE)
▼

Здесь вставляется схема BPMN и текстовое описание того, как процесс работал до внедрения фичи (AS-IS) и как будет работать после (TO-BE).

3. Пользовательские сценарии (Use Cases)
▼

Таблицы Use Case со всеми шагами (Happy Path) и альтернативными ветками.

4. Бизнес-правила и Ограничения
▼

Формулы расчетов, лимиты, права доступа ролей. (Например: скидка не суммируется с баллами).

5. Модель данных (База данных)
▼

ERD схема: Какие таблицы создаем/меняем.

Маппинг данных: Откуда берется каждое поле (Фронт -> API -> БД).

6. Интеграции и Контракты API
▼

Sequence Diagram: Как микросервисы общаются между собой.

REST/Kafka: Ссылки на Swagger, JSON схемы сообщений.

7. UI/UX (Макеты экрана)
▼

Ссылки на Figma и требования к поведению интерфейса (анимации загрузки, маски ввода телефонов).

8. Нефункциональные требования (NFR)
▼

SLA, Нагрузка (RPS), Безопасность, Логирование, Системные требования. Каждый NFR должен быть измеримым (не «система должна быть быстрой», а «p95 ≤ 300 мс при 500 RPS»).

  • Производительность: время ответа, RPS, concurrent users
  • Надёжность: SLA uptime, RTO/RPO, retry-логика
  • Безопасность: схема авторизации, шифрование, логирование аудита
  • Масштабируемость: горизонтальное масштабирование, пиковые нагрузки
  • Совместимость: браузеры, ОС, версии API

Scope — Что входит и что НЕ входит
▼

Один из самых важных разделов — явно фиксирует границы. Предотвращает scope creep и споры «а я думал, это тоже будет».

✅ В скоупе (In Scope)
  • Начисление баллов при оформлении заказа
  • Отображение баланса баллов в личном кабинете
  • API для начисления и списания баллов
❌ Вне скоупа (Out of Scope)
  • Мобильное приложение — в v2.0
  • Интеграция с партнёрскими программами лояльности
  • История операций с баллами — отдельная задача
Глоссарий — единый язык команды
▼

Убирает разночтения. Разработчик и заказчик должны понимать под «Пользователем» одно и то же.

НачислениеДобавление баллов на счёт клиента по факту подтверждённой оплаты заказа

Глоссарий ведётся на уровне проекта (не отдельной фичи) и накапливается со временем.

Раздел 3.2 · Контроль качества
Матрица Трассировки (RTM)
Requirements Traceability Matrix нужна, чтобы ни одно требование не потерялось по пути от бизнеса к тестировщику. Она связывает Бизнес-требования с Функциональными требованиями, модулями кода и Тест-кейсами.
Прямая трассировка (Forward)

От Бизнес-цели к Тест-кейсу. Проверяем: "Мы точно написали код под каждое желание заказчика?".

Обратная трассировка (Backward)

От Тест-кейса или Функции к Бизнес-цели. Проверяем: "А не написали ли разработчики лишнего кода, который бизнес не просил?".

Раздел 3.3 · Приоритизация
Приоритизация требований
Приоритизация — один из самых важных навыков СА. Без неё команда тратит время на «красивые» фичи вместо критически важных. Вигерс: «Если всё — приоритет, значит ничто не приоритет».

MoSCoW — самый популярный метод

Метод из DSDM (Dynamic Systems Development Method). Делит все требования на 4 группы. Главная ценность — помогает PO и заказчику договориться о скоупе спринта или релиза.

🔴
Must Have
Критически важно. Без этого продукт нельзя выпустить. Если не войдёт в релиз — релиза нет.
Пример: авторизация пользователя, оформление заказа, оплата
🟡
Should Have
Важно, но не критично. Должно войти если возможно. Временный обходной путь допустим.
Пример: история заказов, email-уведомления, фильтрация каталога
🟢
Could Have
Желательно, но можно отложить. Войдёт только если осталось время и бюджет.
Пример: тёмная тема, экспорт в PDF, расширенная аналитика
⚫
Won't Have
Явно исключено из текущего скоупа. Фиксируется письменно, чтобы не возвращаться к обсуждению.
Пример: мобильное приложение в v1.0, интеграция с SAP на старте
Ловушка MoSCoW

Заказчики склонны помечать всё как «Must». Правило: в Must не должно входить более 60% от объёма спринта. Если больше — задай вопрос: «Что произойдёт с бизнесом, если мы это не выпустим в первой версии?»

Модель Кано — приоритизация по эмоциям пользователя

Профессор Нориаки Кано предложил классифицировать фичи не по важности, а по тому, как они влияют на удовлетворённость пользователя. Это меняет взгляд на приоритеты: не всё «нужное» одинаково ценно.

Must-Be (Базовые)
Само собой разумеется
  • Пользователь не замечает, когда есть — злится, когда нет
  • Пример: кнопка «Назад», работающий поиск, корзина
  • Вкладывать больше усилий не увеличит удовлетворённость
Performance (Линейные)
Чем больше — тем лучше
  • Удовлетворённость прямо пропорциональна качеству
  • Пример: скорость загрузки, точность поиска, количество фильтров
  • Здесь стоит инвестировать и измерять прогресс
Delighters (Восхитители)
Неожиданные, но любимые
  • Пользователь не ждал — но в восторге
  • Пример: автозаполнение, умные подсказки, персонализация
  • Со временем становятся Must-Be (привыкание)

Матрица «Ценность vs Усилия» — быстрый фильтр беклога

Самый наглядный инструмент для быстрой расстановки приоритетов на воркшопе с командой. Каждое требование помещается в один из квадрантов.

Метод попарного сравнения

Когда требования сложно ранжировать напрямую — сравниваем каждое с каждым. Подходит для небольшого набора (до 10 требований). Результат — объективный рейтинг без споров о «важности».

Как работает

Берёшь каждую пару требований и задаёшь один вопрос: «Что важнее для бизнеса — A или B?» Записываешь победителя. В конце считаешь количество побед у каждого требования — это и есть приоритет. Метод особенно полезен на воркшопах, когда стейкхолдеры не могут договориться.

RICE — количественная приоритизация

Метод от команды Intercom. Убирает субъективность: каждое требование получает числовой балл по четырём факторам. Стандарт в продуктовых компаниях 2024.

🎯
Reach
Сколько пользователей затронет фича за период? Измеряется в людях/транзакциях в месяц.
Пример: 2 000 чел/мес
💥
Impact
Насколько сильно повлияет на пользователя? Шкала: 0.25 / 0.5 / 1 / 2 / 3
3 = массивный эффект
🎲
Confidence
Насколько уверены в оценках? 100% = есть данные. 50% = догадки.
Пример: 80% уверенность
⏱️
Effort
Сколько человеко-месяцев займёт? Чем меньше — тем лучше (делитель).
Пример: 2 чел-мес
Формула и пример расчёта

RICE = (Reach × Impact × Confidence) / Effort

Фича А (умный поиск): (2000 × 2 × 0.8) / 2 = 1600
Фича Б (тёмная тема): (5000 × 0.5 × 1.0) / 1 = 2500
Фича В (push-уведомления): (8000 × 1 × 0.5) / 3 = 1333

Приоритет: Б → А → В. Тёмная тема побеждает не потому что «красиво», а потому что широкий охват и малые затраты.

Управление требованиями
Собрать требования — это половина работы. Вторая половина — управлять ими на протяжении всего проекта: контролировать изменения, отслеживать версии и не допускать хаоса в ТЗ.

Базелайн требований (Requirements Baseline)

Базелайн — это зафиксированная, согласованная версия требований на определённый момент времени. Она становится «точкой отсчёта»: любое изменение после базелайна проходит через официальный процесс.

1
Сбор и документирование
Все требования описаны в SRS / Confluence. Каждое имеет уникальный ID (FR-01, NFR-03).
2
Ревью и согласование
Все стейкхолдеры ознакомились и подписали (или дали аппрув в Jira). Конфликты разрешены.
3
Фиксация базелайна
Документ помечается версией (v1.0) и датой. В Confluence — snapshot, в Git — тег. Это «заморозка».
4
Работа с базелайном
Команда разрабатывает по зафиксированным требованиям. Изменения — только через процесс Change Request.

Процесс управления изменениями (Change Control)

Вигерс настаивает: любое изменение требований после базелайна должно пройти формальный процесс. Это не бюрократия — это защита от «scope creep» (неконтролируемого расползания скоупа).

📝

СА оценивает: какие требования затрагивает изменение, сколько займёт переработка, какие риски. Без этого шага нельзя одобрять.
⚖️
CCB — Совет по контролю изменений

Прототипирование — разбор по Вигерсу

Прототипирование Вигерс называет одной из самых недооценённых техник. Правильный прототип экономит недели переделок кода, потому что проблемы обнаруживаются до написания первой строки.

Throwaway (Одноразовый)
Быстрый черновик для валидации
  • Цель — задать вопрос, получить ответ, выбросить
  • Инструменты: Figma, Balsamiq, даже бумага
  • Занимает часы, а не недели
  • Не привязан к архитектуре или коду
  • Идеален для уточнения UX и бизнес-логики
Evolutionary (Эволюционный)
Прототип, который станет продуктом
  • Начинается как MVP, постепенно обрастает функциями
  • Требования уточняются в процессе разработки
  • Подход используется в Agile / Lean Startup
  • Риск: технический долг если «временное» стало постоянным
  • Нужна дисциплина рефакторинга
Практика: Lo-Fi vs Hi-Fi прототип

Lo-Fi (бумага / wireframe): 30 минут, рисуешь прямоугольники и стрелки. Используй когда нужно быстро проверить логику сценария или расположение элементов. Hi-Fi (Figma с компонентами): несколько часов, кликабельный прототип. Используй когда нужно показать заказчику «почти готовый» интерфейс или протестировать с реальными пользователями.

User Story и Use Case
User Story (Agile)
Как [Роль], я хочу [Действие], чтобы [Ценность]

Используется в продуктовых командах. Коротко, ясно бизнесу, дополняется критериями приемки (Acceptance Criteria).

Детальный пошаговый сценарий в табличном виде.

  • Предусловие: Юзер залогинен.

Job Story — альтернатива User Story

Job Story появилась как ответ на главный недостаток User Story — роль слишком абстрактна и не передаёт контекст ситуации. Формат разработан в Intercom.

User Story
Как зарегистрированный пользователь, я хочу получить уведомление об оплате, чтобы знать что платёж прошёл.

Роль размыта. Непонятно в какой ситуации это нужно и почему именно сейчас.

Job Story
Когда я только что оплатил заказ, я хочу сразу получить подтверждение, чтобы не беспокоиться что деньги списались а заказ не создался.

Конкретная ситуация. Понятна мотивация. Можно сразу придумать решение.

Формула Job Story

Когда [ситуация/триггер] — конкретный момент когда возникает потребность
Я хочу [мотивация/действие] — что пользователь хочет сделать
Чтобы [ожидаемый результат] — какую проблему это решает

Когда использовать Job Story вместо User Story

Job Story лучше подходит когда важен контекст ситуации — не кто пользователь, а когда и почему у него возникает потребность. Особенно хорош для сложных продуктов где один и тот же пользователь имеет разные мотивации в разных ситуациях.

Раздел 6 · Приёмка
Acceptance Criteria — полный разбор
AC — это контракт между аналитиком и командой. Без них разработчик делает «как понял», QA не знает что тестировать, а PO не знает как принять работу.
Главный принцип

Хороший AC — это исполняемая спецификация. Можно написать автотест прямо по нему. Если AC нельзя автоматизировать — он написан плохо.

Формат Given / When / Then — детально

ЧастьВопросЧто писатьОшибки

AC-1 Happy Path — успешный вход
▼

Given пользователь зарегистрирован, аккаунт активен, находится на /login

When вводит корректный email и пароль, нажимает «Войти»

Then перенаправляется на /dashboard за < 1 сек

And в хедере отображается имя пользователя

And в куки записывается httpOnly JWT-токен

And создаётся запись в таблице session_log

AC-2 Негативный — неверный пароль
▼

Given пользователь на странице /login

When вводит корректный email, но неверный пароль

Then система показывает ошибку «Неверный email или пароль» (формулировка без уточнения — безопасность)

And поле пароля очищается, email сохраняется

And счётчик неверных попыток увеличивается на 1

And страница НЕ перезагружается

AC-3 Граничный — блокировка после 5 попыток
▼

Given пользователь ввёл неверный пароль 5 раз подряд

When делает 6-ю попытку входа

Then аккаунт блокируется на 15 минут

And показывается сообщение «Аккаунт временно заблокирован. Попробуйте через 15 минут»

And на email приходит уведомление о подозрительной активности

And кнопка «Войти» становится неактивной до истечения таймера

AC-4 Edge case — несуществующий email
▼

▼

When система обрабатывает запросы

Then время ответа POST /auth/login ≤ 300ms для 95% запросов (p95)

And время ответа ≤ 1000ms для 99% запросов (p99)

And error rate < 0.1%

Признаки плохого AC — чек-лист

❌ Плохо✅ Как исправить
«Система должна работать быстро»«Время ответа API ≤ 300ms при нагрузке 500 RPS (p95)»
«Пользователь видит ошибку»«Система показывает inline-ошибку под полем: "Введите корректный email"»
«Данные сохраняются в БД»«В таблице users создаётся запись с полями name, email, created_at; password хранится как bcrypt-хэш»
«Все поля обязательны»Отдельный AC для каждого обязательного поля с конкретным текстом ошибки
«Форма валидируется»AC для каждого правила валидации: формат email, длина пароля ≥8, спецсимвол обязателен
«Работает как раньше»Описать конкретное поведение — «как раньше» не тестируется

Definition of Done (DoD) — стандарт готовности команды

Типичный DoD в Scrum-команде:

  • ✅ Код написан и прошёл Code Review (≥ 2 апрувера)
  • ✅ Unit-тесты написаны, покрытие ≥ 80%
  • ✅ Integration-тесты пройдены
  • ✅ QA проверил по тест-кейсам на staging
  • ✅ Нет критических и высоких дефектов
  • ✅ Swagger/OpenAPI обновлён
  • ✅ Changelog и документация обновлены
  • ✅ Feature flag настроен (если нужен)
  • ✅ PO принял демо
  • ✅ Метрики/алерты настроены (для NFR)
AC vs DoD — в чём разница

AC — условия конкретной User Story. Разные для каждой задачи. Написаны аналитиком.

DoD — стандарт качества для ВСЕЙ команды. Одинаковый для всех задач. Написан командой и не меняется от задачи к задаче.

Задача закрывается только когда выполнены И AC И DoD.

Главная ошибка

Путать AC и DoD. AC — «что делает фича». DoD — «как мы убеждаемся что сделали это качественно».

Моделирование бизнес-процессов AS-IS и TO-BE. Понятна и бизнесу, и разработчикам. Показывает кто, что и в каком порядке делает, с ветвлениями и параллельными потоками.

Правила хорошего BPMN

Каждый gateway должен закрываться таким же типом. XOR открыл → XOR закрыл. Task всегда с глаголом. Один конец = один End Event (если несколько — используй разные End Events с иконками).

Антипаттерны

Смешивать XOR и AND для одного gateway. Писать в Task «Заказ» вместо «Обработать заказ». Делать один огромный процесс вместо Sub-Process.

C4 Model — когда применять

Архитектурные диаграммы с 4 уровнями детализации. L1 — для заказчика (система в целом), L2 — для команды (контейнеры), L3 — для разработчиков (компоненты), L4 — код. СА работает с L1 и L2.

L2 ContainerВся командаТехнические блоки: веб-приложение, БД, API, очередьContainer (Web App, API, DB, Queue)

Каждый элемент подписывается по схеме: Название [технология] + краткое описание. Пример: «Order Service [Spring Boot] — управляет созданием и состоянием заказов»

Показывает как движутся данные через систему — от источников через процессы к хранилищам. Не показывает порядок (для этого Sequence) и не показывает ответственных (для этого BPMN).

Процесс должен иметь минимум один вход и один выход. Внешние сущности не общаются между собой напрямую. Потоки именуются.

Антипаттерны

Хранилище-хранилище без процесса между ними. Процесс только с выходом (откуда данные?). Безымянные потоки.

Контекстная диаграмма (System Context Diagram)

Один из первых артефактов по Вигерсу — диаграмма контекста системы. Показывает систему как «чёрный ящик» в окружении внешних сущностей, которые с ней взаимодействуют. Отвечает на вопрос: «Что находится внутри системы, а что — снаружи?»

Зачем нужна контекстная диаграмма

Контекстная диаграмма — это граница системы. Она фиксирует: кто является пользователями системы, какие внешние системы интегрируются, какие данные входят и выходят. Без неё часто возникают споры: «А это вообще наша ответственность?»

Таблица решений (Decision Table)

Таблица решений — мощный инструмент для формализации сложной бизнес-логики с множеством условий. Вигерс рекомендует использовать её всегда, когда в требовании есть больше 3 условий «если… то…». Она однозначна, легко тестируется и не допускает пропущенных сценариев.

Когда использовать Decision Table

Если видишь текст вида: «Если клиент VIP и сумма больше 5000 или у него есть промокод и это не распродажа, то...» — это сигнал. Перепиши это как таблицу решений, иначе разработчик и тестировщик поймут это по-разному.

Раздел 8 · Бизнес-процессы
Моделирование процессов: AS-IS → TO-BE → GAP
Ключевой навык СА — описать как работает процесс сейчас, как должен работать после внедрения, и зафиксировать разрыв как список требований.
AS-IS — «Как есть»
Текущее состояние
  • Описывает существующий процесс без прикрас
  • Выявляет узкие места, ручные шаги, дубликаты
  • Собирается через интервью и наблюдение
  • Часто шокирует — «мы так работаем?!»
TO-BE — «Как должно быть»
Целевое состояние
  • Описывает процесс после внедрения системы
  • Устраняет выявленные проблемы AS-IS
  • Проектируется совместно с бизнесом
  • Становится основой для требований

Сквозной пример: Обработка заявки на кредит

AS-IS — как сейчас (ручной процесс)
▼
1
Клиент приходит в отделение
Заполняет бумажную анкету от руки. Время: 30-40 минут
2

Выявленные проблемы AS-IS

🔴 Ошибки ручного ввода (~15% заявок с ошибками)

🔴 Долгое ожидание (3-5 дней) — клиенты уходят к конкурентам

🔴 Нет трекинга статуса — клиент не знает что происходит

🔴 Нет онлайн-канала — только физическое присутствие

🔴 Аналитик тратит 70% времени на рутину (звонки в БКИ)

▼

✅ Ошибок ввода нет — валидация на форме

✅ Клиент видит статус онлайн в любой момент

✅ Канал доступен 24/7 без физического присутствия

✅ Аналитик занимается только сложными кейсами

Метрики процесса — как измерить улучшение

Lead Time
Полное время от старта до результата. AS-IS: 3-5 дней. TO-BE: 5 мин — 1 час.
Cycle Time
Время активной работы без ожидания. Ручной ввод (15 мин) → автоматически (0).
Throughput
Кол-во заявок в день. AS-IS: 50/день (один отдел). TO-BE: 500+/день (автоматизация).
Error Rate
Доля ошибочных записей. AS-IS: 15%. TO-BE: < 1% (только системные ошибки).

BPMN TO-BE — визуализация нового процесса

Моделирование · Интерфейсы
UI и Прототипирование
СА не рисует дизайн — но обязан уметь описать интерфейс, создать прототип для согласования и поставить задачу дизайнеру точно и без разночтений.
Зачем СА уметь работать с UI

Когда требования написаны текстом, каждый представляет интерфейс по-своему. Прототип — это визуальный договор: заказчик видит что получит, разработчик понимает что строить, дизайнер знает ограничения. Прототип экономит недели переделок.

Роль СА в UI проектировании

Что делает СА
  • Создаёт wireframes — схематичные макеты без дизайна
  • Описывает user flow — путь пользователя по экранам
  • Фиксирует логику поведения элементов (что происходит при клике)
  • Согласовывает прототип с заказчиком до передачи дизайнеру
  • Создаёт ISD / карту экранов системы
Что делает дизайнер (UX/UI)
  • Создаёт визуальный дизайн — цвета, шрифты, стили
  • Проводит пользовательское тестирование
  • Разрабатывает дизайн-систему
  • Анимации и микроинтеракции
  • Работает в Figma на уровне пикселей

Персоны пользователей

Персона — архетипический портрет типичного пользователя. Помогает всей команде думать о конкретном человеке, а не абстрактном «юзере».

Базы Данных (SQL vs NoSQL)
Реляционные (SQL)
  • Хранят данные в строгих таблицах (строки и столбцы).
  • Гарантируют транзакции (ACID).
  • Используются в финансах и e-commerce (PostgreSQL, MySQL, Oracle).
Нереляционные (NoSQL)
  • Хранят документы (JSON), Key-Value, графы.
  • Гибкая схема (BASE). Отлично масштабируются горизонтально.
  • Используются для Big Data, кэша, соцсетей (MongoDB, Redis, Cassandra).

Так аналитики визуализируют архитектуру сотен таблиц.

Свойства ACID (Фундамент банковских систем)

  • Atomicity (Атомарность): Транзакция выполняется целиком или не выполняется вообще (откат).
  • Consistency (Консистентность): Транзакция не нарушает правила БД (нельзя уйти в минус на дебетовой карте).
  • Isolation (Изолированность): Параллельные транзакции не мешают друг другу.
  • Durability (Долговечность): Если БД ответила "Успех", данные записаны на диск и переживут отключение света.
Раздел 9.1 · Архитектура БД
Нормализация и Индексы
То, что необходимо для проектирования БД, которая не "ляжет" под нагрузкой.

Нормальные формы (Нормализация)

Избавление базы данных от дублирования и аномалий обновления.

1NF (Первая нормальная форма)
▼

Каждая ячейка таблицы неделима. Нельзя хранить в колонке `telephones` строку "89991234567, 89997654321". Нужно выносить телефоны в отдельные строки.

2NF (Вторая нормальная форма)
▼

Таблица находится в 1NF, и все неключевые столбцы зависят от всего первичного ключа, а не от его части. (Актуально, если Primary Key составной).

3NF (Третья нормальная форма)
▼

Таблица находится в 2NF, и неключевые столбцы не зависят друг от друга. Если у нас есть поля `price`, `quantity` и `total_sum`, то `total_sum` нужно удалить из таблицы, так как оно вычисляется из первых двух.

Индексы в БД (Как ускорить SELECT)

B-Tree (Дерево)
  • Используется по умолчанию в SQL.
  • Отлично работает для условий `=`, `>`, `<`, `BETWEEN`.
  • Важно: Поиск `LIKE '%текст'` не использует B-Tree индекс! Индекс работает только при `LIKE 'текст%'` (поиск по префиксу).
Hash Индекс
  • Использует хэш-функцию.
  • Работает быстрее B-Tree, но **только** для строгого равенства (`=`).
  • Не умеет искать диапазоны (>, <).

Порядок выполнения SQL-запроса

WHERE vs HAVING — главный вопрос на собесе

WHERE фильтрует строки до группировки — работает с сырыми данными.
HAVING фильтрует после группировки — работает с агрегатами (COUNT, SUM, AVG).
Правило: если в условии есть агрегатная функция — используй HAVING.

JOIN — визуальная шпаргалка

Агрегации (GROUP BY & HAVING)

-- Клиенты с суммой заказов > 10 000 и количеством > 2 SELECT u.user_id, u.name, COUNT(o.order_id) AS orders_count, SUM(o.total_amount) AS total_spent, AVG(o.total_amount) AS avg_check FROM users u LEFT JOIN orders o ON u.user_id = o.user_id WHERE o.status = 'completed' -- фильтр ДО группировки GROUP BY u.user_id, u.name HAVING SUM(o.total_amount) > 10000 -- фильтр ПОСЛЕ группировки AND COUNT(o.order_id) > 2 ORDER BY total_spent DESC;

Оконные функции — без схлопывания строк

-- ROW_NUMBER, RANK, LAG — топ фич для аналитика SELECT user_id, order_id, total_amount, -- Нумерация заказов по каждому клиенту ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at) AS order_num, -- Нарастающая сумма покупок SUM(total_amount) OVER (PARTITION BY user_id ORDER BY created_at) AS running_total, -- Сумма предыдущего заказа (для сравнения) LAG(total_amount) OVER (PARTITION BY user_id ORDER BY created_at) AS prev_order FROM orders;

📝 Практика — реши сам

1
Найти топ-3 категории товаров по выручке за последние 30 дней
Таблицы: orders(order_id, created_at, status), order_items(order_id, product_id, qty, price), products(product_id, name, category). Учитывать только заказы со статусом 'completed'.
👁 Показать ответ
SELECT p.category, SUM(oi.qty * oi.price) AS revenue FROM orders o JOIN order_items oi ON o.order_id = oi.order_id JOIN products p ON oi.product_id = p.product_id WHERE o.status = 'completed' AND o.created_at >= NOW() - INTERVAL '30 days' GROUP BY p.category ORDER BY revenue DESC LIMIT 3;
2
Найти пользователей, которые зарегистрировались, но не сделали ни одного заказа
Таблицы: users(user_id, name, email, created_at), orders(order_id, user_id, created_at).
👁 Показать ответ
-- Вариант 1: LEFT JOIN + IS NULL SELECT u.user_id, u.name, u.email FROM users u LEFT JOIN orders o ON u.user_id = o.user_id WHERE o.order_id IS NULL; -- Вариант 2: NOT EXISTS (быстрее на больших таблицах) SELECT user_id, name, email FROM users u WHERE NOT EXISTS ( SELECT 1 FROM orders o WHERE o.user_id = u.user_id );
3
Посчитать процент выручки каждого менеджера от общей
Таблица: sales(sale_id, manager_name, amount). Нужен столбец с процентом.
👁 Показать ответ
SELECT manager_name, SUM(amount) AS manager_revenue, ROUND( SUM(amount) * 100.0 / SUM(SUM(amount)) OVER(), -- оконная функция по всем строкам 2 ) AS pct_of_total FROM sales GROUP BY manager_name ORDER BY manager_revenue DESC;

CTE — Common Table Expressions (WITH)

CTE позволяет дать имя подзапросу и обращаться к нему как к временной таблице. Делает сложные запросы читаемыми. Аналог «переменной» в SQL.

WITH top_customers AS ( SELECT customer_id, SUM(amount) AS total_spent FROM orders WHERE status = 'PAID' GROUP BY customer_id HAVING SUM(amount) > 10000 ), customer_orders AS ( SELECT c.name, tc.total_spent, COUNT(o.id) AS orders_count FROM top_customers tc JOIN customers c ON c.id = tc.customer_id JOIN orders o ON o.customer_id = tc.customer_id GROUP BY c.name, tc.total_spent ) SELECT * FROM customer_orders ORDER BY total_spent DESC;

CTE выполняются сверху вниз. Каждый может ссылаться на предыдущий. Итоговый SELECT работает со всеми определёнными CTE. Это мощнее и читаемее цепочки подзапросов.

UNION vs UNION ALL — частая ошибка

UNION
Убирает дубликаты
  • Выполняет DISTINCT на объединённых строках
  • Медленнее — нужна сортировка для поиска дублей
  • Использовать когда дубликаты реально возможны и нежелательны
SELECT email FROM users_2023 UNION SELECT email FROM users_2024; -- вернёт уникальные email из обоих лет
UNION ALL
Оставляет все строки
  • Просто склеивает результаты без проверки дублей
  • Быстрее — нет лишней сортировки
  • Использовать когда дубликаты допустимы или их нет по логике
SELECT amount FROM sales_q1 UNION ALL SELECT amount FROM sales_q2; -- все транзакции за оба квартала
Правило выбора

По умолчанию используй UNION ALL — он быстрее. Переходи на UNION только если точно знаешь, что в источниках могут быть одинаковые строки и тебе нужна уникальность.

Раздел 11 · Архитектура
Монолит vs Микросервисы
Самый частый вопрос на собесах уровня Middle. СА должен понимать не только разницу, но и когда что применять и какие проблемы возникают при переходе.
⚡ Правило большого пальца

Команда < 10 чел. или MVP → Монолит. Команда > 15 чел., много доменов, нужен независимый деплой → Микросервисы. Никогда не начинай с микросервисов только потому что «Netflix так делает».

Модульный монолит — золотая середина

Что это такое?

Один деплой как в монолите, но код строго разделён на независимые модули с чёткими API между ними. Модули не могут обращаться к чужим таблицам в БД напрямую — только через публичный интерфейс модуля.

Когда использовать: растущий стартап, который ещё не вырос до масштаба микросервисов, но хочет сохранить порядок в коде. Правильный модульный монолит легко «распилить» на микросервисы позже.

Как правильно переходить с монолита на микросервисы

1
Strangler Fig Pattern — «удушающий инжир»
Не переписываешь монолит сразу. Рядом поднимаешь новый микросервис и постепенно перенаправляешь трафик через API Gateway. Монолит «отмирает» постепенно, как дерево, обвитое лианой.
2
Выдели самый независимый домен первым
Сначала уведомления (Email/SMS) — они почти ни от чего не зависят. Потом отчётность. В последнюю очередь — ядро с заказами и деньгами.
3
Один сервис — одна БД
Главное правило: запрещено дать двум сервисам доступ к одной таблице. Если сервис А нужны данные сервиса Б — он запрашивает их через REST API сервиса Б.
4
Введи наблюдаемость (Observability)
Без централизованного логирования (ELK Stack) и трассировки (Jaeger) поддерживать микросервисы — кошмар. Это не «потом», это с первого дня.

Роль СА при проектировании архитектуры

Что делает СА
  • Описывает границы доменов (Domain Boundaries)
  • Определяет, какие данные нужны каждому сервису
  • Проектирует контракты API между сервисами
  • Документирует Sequence Diagram для каждого сценария
  • Согласует с командой форматы событий в Kafka
Что решает Архитектор (не СА)
  • Выбор языков программирования
  • Инфраструктура (Kubernetes, Docker Swarm)
  • Сетевые настройки, Service Mesh (Istio)
  • Стратегия деплоя (Blue/Green, Canary)

Событийно-ориентированная архитектура (EDA)

В EDA микросервисы не вызывают друг друга напрямую (синхронно). Вместо этого они генерируют События (Events) в шину (Kafka/RabbitMQ). Другие сервисы подписываются на эти события.

Пример: Когда пользователь оплачивает заказ, Сервис Оплат не отправляет запрос в Сервис Уведомлений. Он просто кидает событие "order.paid". Сервис Уведомлений сам его читает и шлёт СМС, а Сервис Склада читает и начинает сборку. Это обеспечивает слабую связанность (Loose Coupling).

API Gateway & BFF

API Gateway (Шлюз) — единая точка входа. Принимает запрос, проверяет JWT-токен, логирует и маршрутизирует в нужный микросервис. Клиент не знает, к каким именно сервисам обращается.

BFF (Backend For Frontend) — паттерн, когда под каждый клиент (Web, Mobile, Smart TV) делается свой шлюз, возвращающий только нужные для конкретного экрана данные. Мобилке не нужен тяжёлый JSON для десктопа.

Масштабирование

Вертикальное (Scale UP)

Покупка более мощного сервера (больше RAM, CPU). Минус: есть физический предел, сервер всё ещё один — точка отказа.

Горизонтальное (Scale OUT)

Добавление новых серверов в кластер (было 2, стало 10). Плюс: практически бесконечный рост, устойчивость к падениям нод.

Закон Конвея (Conway's Law)

Один из самых важных принципов в архитектуре, о котором часто забывают на уровне требований.

Формулировка

«Организации проектируют системы, которые копируют коммуникационные структуры этих организаций» — Мелвин Конвей, 1967. Актуально как никогда в 2024.

Антипаттерн
Архитектура не совпадает со структурой команд
  • Одна команда владеет 5 разными микросервисами
  • Три команды совместно владеют одним сервисом
  • Результат: бесконечные согласования, медленные релизы, межкомандные конфликты
Правильно: Inverse Conway Maneuver
Сначала спроектируй команды, потом архитектуру
  • Одна команда — один сервис (или тесно связанная группа)
  • Чёткие границы ответственности = чёткие API-контракты
  • Spotify Model, Amazon «Two-Pizza Teams»
  • Команда может релизить независимо — без согласований
Практическое применение для СА

Когда проектируешь микросервисную архитектуру — спроси: «Сколько команд будет разрабатывать это?» Если три команды разрабатывают один монолитный сервис — архитектура будет воспроизводить конфликты между ними в коде. Граница микросервиса должна совпадать с границей команды. Это не технический вопрос — это организационный.

Раздел 11.1 · Системный Дизайн
Паттерны микросервисов
Разрезав монолит на микросервисы, мы получаем новые проблемы: сеть может упасть, а транзакция может застрять на полпути. Эти паттерны решают такие проблемы.
⚡
Circuit Breaker (Предохранитель)
Если сторонний API (например, Банк) начинает жестко тормозить, наш сервис тоже зависнет в ожидании ответа. Предохранитель "размыкает" цепь: если 5 запросов подряд упали с таймаутом, мы временно сразу отдаем ошибку клиенту, не пытаясь дергать банк, чтобы не уронить систему окончательно.
🔄
Паттерн Saga (Сага)
Заменяет ACID транзакции в распределённых системах. Если операция в одном из сервисов провалилась — Сага запускает «компенсирующие транзакции» в уже выполненных шагах для отката.
📦
Transactional Outbox
Проблема: мы сохранили заказ в нашу БД, но брокер Kafka упал, и мы не смогли отправить событие. Решение: вместе с заказом в той же транзакции БД пишем событие в таблицу `Outbox`. Отдельный скрипт читает эту таблицу и гарантированно досылает события в Kafka.

Saga: Оркестрация vs Хореография

Это два принципиально разных подхода к реализации одного паттерна. Выбор между ними — одно из ключевых архитектурных решений при проектировании распределённых транзакций.

Orchestration (Оркестрация)
Есть центральный дирижёр (Orchestrator)
  • Один сервис-оркестратор знает весь сценарий и командует остальными
  • Сервисы не знают друг о друге — только об оркестраторе
  • Легко трейсить: весь flow в одном месте
  • Риск: оркестратор становится единой точкой отказа и «богом»
  • Инструменты: Temporal.io, AWS Step Functions, Camunda
Orchestrator → OrderService: создай заказ Orchestrator → PaymentService: спиши деньги PaymentService → Orchestrator: ошибка! Orchestrator → OrderService: отмени заказ ← компенсация
Choreography (Хореография)
Каждый сервис реагирует на события
  • Нет центрального координатора — сервисы слушают события в брокере
  • Каждый сервис знает: «при событии X я делаю Y и публикую Z»
  • Слабая связность — сервисы полностью независимы
  • Риск: сложно понять общую картину flow, сложнее дебажить
  • Инструменты: Kafka, RabbitMQ события
OrderService → [order.created] → Kafka PaymentService слушает → списывает деньги PaymentService → [payment.failed] → Kafka OrderService слушает → отменяет заказ ← компенсация
Когда что выбирать

Оркестрация — когда flow сложный, много шагов, нужна видимость. Хореография — когда сервисов мало, flow простой, нужна максимальная независимость команд. На практике крупные системы часто комбинируют: локальная хореография внутри домена + оркестрация на уровне cross-domain flow.

Идемпотентность API (Важный концепт)

Что такое Идемпотентность?

Операция идемпотентна, если её многократный вызов даёт тот же результат системы, что и один вызов.

Зачем это нужно? Если у клиента "моргнул" интернет в момент оплаты, он может случайно отправить запрос дважды. Если метод идемпотентен, деньги спишутся только один раз. Для этого в заголовках часто передают Idempotency-Key (уникальный UUID операции).

GraphQL и gRPC
Два современных протокола которые активно вытесняют REST в определённых задачах. СА должен понимать когда и зачем их применяют.

Почему не только REST

Ограничения REST

REST отлично работает для простых CRUD-операций. Но у него есть проблемы: overfetching — клиент получает больше данных чем нужно; underfetching — нужно делать несколько запросов чтобы собрать нужные данные; версионирование API — сложно менять схему без поломки клиентов. GraphQL и gRPC решают эти проблемы каждый по-своему.

GraphQL

GraphQL — язык запросов для API, разработанный Facebook в 2015. Клиент сам описывает какие данные ему нужны — и получает ровно их, не больше и не меньше.

Как работает GraphQL
  • Один endpoint: /graphql
  • Клиент пишет запрос (query) с нужными полями
  • Сервер возвращает ровно те поля которые запросили
  • Можно запросить данные из нескольких «таблиц» за один запрос
  • Есть mutations (изменение данных) и subscriptions (real-time)
Когда применяется
  • Мобильные приложения — нужно минимизировать трафик
  • Сложные продукты с гибкими экранами (разные клиенты — разные данные)
  • BFF (Backend For Frontend) паттерн
  • Когда несколько команд работают с одним API
  • Примеры: GitHub API v4, Shopify, Facebook
Пример GraphQL запроса vs REST

REST (3 запроса): GET /users/42 → GET /users/42/orders → GET /orders/7/items

GraphQL (1 запрос):

query {
  user(id: 42) {
    name
    email
    orders(last: 5) {
      id
      total
      items { name quantity }
    }
  }
}

gRPC

gRPC — фреймворк удалённого вызова процедур от Google (2016). Использует Protocol Buffers (protobuf) для сериализации данных вместо JSON. Работает поверх HTTP/2.

Как работает gRPC
  • Контракт описывается в .proto файле (не JSON, не YAML)
  • Данные передаются в бинарном формате — быстрее и компактнее JSON
  • Поддерживает streaming в обе стороны
  • Типизация — ошибки типов находятся на этапе компиляции
  • Из .proto автогенерируются клиенты на любом языке
Когда применяется
  • Коммуникация между микросервисами внутри системы
  • Высоконагруженные сервисы где важна скорость
  • Streaming данных (чат, телеметрия, видео)
  • Внутренние API (не публичные)
  • Примеры: внутренние сервисы Google, Uber, Netflix
Пример .proto файла
syntax = "proto3";

service OrderService {
  rpc GetOrder (OrderRequest) returns (OrderResponse);
  rpc ListOrders (ListRequest) returns (stream OrderResponse);
}

message OrderRequest { int32 order_id = 1; }
message OrderResponse {
  int32 id = 1;
  string status = 2;
  double total = 3;
}

Архитектура · Проектирование
DDD — Domain-Driven Design
Предметно-ориентированное проектирование — подход где структура системы отражает структуру бизнеса. СА использует DDD чтобы говорить с разработчиками и бизнесом на одном языке.
Зачем СА знать DDD

DDD — это не про код, это про общий язык и границы ответственности. Когда бизнес говорит «заказ», разработчик может иметь в виду совсем другую сущность. DDD учит фиксировать единый глоссарий и чётко разграничивать зоны системы. Именно это и делает СА.

Ключевые концепции DDD

1
Ubiquitous Language Единый язык
Общий словарь терминов который используют и бизнес, и разработчики, и аналитики. Один термин — одно значение во всей команде.
Пример проблемы без единого языка

Бизнес говорит «клиент» — имея в виду юрлицо.
Разработчик слышит «клиент» — создаёт таблицу users для физлиц.
СА с DDD: фиксирует в глоссарии: Клиент = юридическое лицо, Пользователь = сотрудник клиента.

2
Bounded Context Ограниченный контекст
Чёткая граница внутри которой термины и модели имеют однозначное значение. Разные контексты могут использовать одно слово по-разному — и это нормально.
Контекст «Продажи»

«Заказ» = намерение купить, может быть отменён, содержит позиции и скидки

Контекст «Доставка»

«Заказ» = физическая посылка с адресом, весом, статусом доставки

Одно слово — разный смысл в разных контекстах. DDD говорит: это нормально, главное чётко зафиксировать границы.

3
Context Map Карта контекстов
Диаграмма показывающая все Bounded Context системы и отношения между ними. Главный артефакт DDD для СА.
4
Aggregate Агрегат
Группа связанных объектов которая обрабатывается как единое целое. У каждого агрегата есть корневой объект (Aggregate Root) через который происходит всё взаимодействие.
Пример агрегата «Заказ»

Aggregate Root: Заказ (Order) — только через него можно изменить состав
Entities внутри: Позиция заказа (OrderItem), Адрес доставки
Value Objects: Сумма, Валюта, Статус

Нельзя напрямую изменить OrderItem — только через Order. Это защищает бизнес-инварианты.

Брокеры: Kafka, RabbitMQ и SOAP
Брокеры сообщений — основа асинхронной архитектуры. SOAP — стандарт legacy и корпоративных интеграций. Оба темы частых вопросов на собесах.
Зачем нужны брокеры?

Если Сервис Корзины вызывает Склад синхронно по REST — и Склад тормозит, Корзина тоже висит. С брокером: Корзина кидает событие order.created и сразу отвечает пользователю «Ок». Склад заберёт событие когда освободится. Сервисы независимы.

Apache Kafka — детальный разбор

Ключевые концепции Kafka
  • Topic — именованный журнал событий. Аналог таблицы в БД
  • Partition — Topic разбит на партиции для параллелизма
  • Offset — позиция сообщения в партиции. Consumer помнит свой offset
  • Producer — публикует сообщения в Topic
  • Consumer — читает сообщения из Topic
  • Consumer Group — группа Consumer'ов. Каждое сообщение читает только один из группы
  • Retention — время хранения сообщений (default 7 дней). Сообщения НЕ удаляются после прочтения
  • Replication Factor — кол-во копий данных на разных брокерах (обычно 3)
Когда использовать Kafka
  • Одно событие нужно нескольким потребителям независимо
  • Нужна история событий (Event Sourcing, аудит)
  • Высокий throughput (миллионы событий в секунду)
  • Стриминговая аналитика в реальном времени
  • Логирование и мониторинг
Пример

order.paid → читают: Склад (резервирует) + Loyalty (начисляет баллы) + Analytics (считает выручку) + Email-сервис (шлёт чек)

⚠️ Порядок сообщений в Kafka

Kafka гарантирует порядок только внутри одной партиции. Чтобы события одного заказа шли по порядку (Создан → Оплачен → Доставлен) — укажи Message Key = order_id. Тогда все события этого заказа попадут строго в одну партицию.

RabbitMQ — детальный разбор

Ключевые концепции RabbitMQ
  • Exchange — принимает сообщения и маршрутизирует в Queue
  • Queue — буфер сообщений для Consumer'а
  • Binding — правило связи Exchange → Queue
  • Routing Key — метка сообщения для маршрутизации
  • ACK — подтверждение обработки от Consumer
  • DLQ (Dead Letter Queue) — очередь для сообщений которые не удалось обработать N раз
Типы Exchange в RabbitMQ
  • Fanout — рассылает всем подключённым Queue (broadcast)
  • Direct — доставляет в Queue с точным совпадением routing key
  • Topic — маршрутизация по паттерну: order.*, *.created
  • Headers — маршрутизация по заголовкам сообщения
Когда RabbitMQ

Одна задача — один исполнитель (рассылка email). Сложная маршрутизация. Нужны приоритеты очередей.

DLQ — Dead Letter Queue (обязательно знать)

Что это и зачем

Если Consumer не может обработать сообщение (ошибка валидации, внешний сервис недоступен) — сообщение не теряется. После N попыток оно попадает в Dead Letter Queue.

Из DLQ: дежурный инженер разбирает ошибки, исправляет данные и пускает сообщения на повторную обработку. Это критически важно для финансовых систем.

SOAP — интеграции с legacy и корпоративными системами

Почему SOAP важно знать в 2025?

22% вакансий СА явно упоминают SOAP. Банки, страховые, госсистемы (ФНС, Госуслуги, СМЭВ), медицина — там десятилетиями работает SOAP и мигрировать не планируют. Без базового понимания SOAP ты закрыт для этих рынков.

SOAP
Протокол (не просто формат)
  • Только XML — никакого JSON
  • Один эндпоинт, разные операции через Action
  • WSDL — машиночитаемый контракт сервиса
  • WS-Security — шифрование и подписи на уровне протокола
  • Строгая типизация — XSD-схема
  • ACID-транзакции через WS-AtomicTransaction
REST vs SOAP — ключевые отличия
  • Формат: REST = JSON/XML. SOAP = только XML
  • Контракт: REST = OpenAPI (опционально). SOAP = WSDL (обязательно)
  • Транспорт: REST = только HTTP. SOAP = HTTP, SMTP, FTP
  • Ошибки: REST = HTTP коды. SOAP = SOAP Fault в теле
  • Типизация: REST = слабая. SOAP = строгая XSD
  • Когда: SOAP = корпорации, банки, госсистемы, сложная безопасность

Access Token + Refresh Token — правильная схема JWT

Короткоживущий Access Token решает проблему безопасности, но создаёт UX-проблему: пользователь вынужден перелогиниваться каждые 15 минут. Решение — пара токенов.

Access Token
Краткосрочный (15 мин — 1 час)
  • Передаётся в каждом запросе в заголовке Authorization: Bearer ...
  • Содержит роль и user_id — сервер не обращается к БД
  • При компрометации живёт недолго → ущерб минимален
  • Хранить в памяти (переменная JS), не в localStorage
Refresh Token
Долгосрочный (7–30 дней)
  • Используется только для получения нового Access Token
  • Хранится в HttpOnly cookie — JS не может прочитать
  • Сервер хранит в БД → можно отозвать при logout
  • Rotation: при каждом обновлении выдаётся новый
1
Логин → сервер выдаёт пару токенов
Access Token (15 мин) + Refresh Token (30 дней). Refresh → в HttpOnly cookie, Access → в память приложения.
2
Запросы к API с Access Token
Пока Access Token жив — обычная работа. Сервер верифицирует подпись без обращения к БД.
3
Access Token истёк → тихое обновление
Приложение автоматически обращается к /auth/refresh с Refresh Token. Сервер проверяет его в БД и выдаёт новую пару. Пользователь ничего не замечает.
4
Logout → отзыв Refresh Token
Сервер удаляет Refresh Token из БД. Access Token доживает свои 15 минут, но новый уже не выдастся.

OAuth 2.0 — «Войти через Google» шаг за шагом

OAuth 2.0 — это протокол делегирования доступа. Пользователь даёт приложению «А» разрешение получить данные из приложения «Б» (Google, VK, Яндекс) без передачи пароля.

1
👤 Пользователь → Ваш сайт
Нажимает кнопку «Войти через Google»
2
🌐 Ваш сайт → Google
Редирект на Google с параметрами: client_id, redirect_uri, scope=email profile, state=random_csrf_token
3
👤 Пользователь → Google
Видит диалог «Приложение X хочет получить доступ к вашему email». Нажимает «Разрешить»

OpenID Connect — OAuth 2.0 + аутентификация

OAuth 2.0 решает задачу авторизации (доступ к ресурсам), но не аутентификации (кто этот человек?). OpenID Connect (OIDC) — это тонкий слой поверх OAuth 2.0, который добавляет стандартизированную аутентификацию.

OAuth 2.0
«Можно ли приложению X получить данные?»
  • Выдаёт access_token для доступа к API
  • Не говорит кто пользователь — только что он разрешил доступ
  • Нет стандарта для профиля пользователя
OpenID Connect
«Кто этот пользователь + можно ли доступ?»
  • Добавляет id_token (JWT с данными пользователя)
  • Стандартный endpoint /userinfo → email, name, sub
  • «Войти через Google/VK/Яндекс» — это OIDC, не чистый OAuth
Как запомнить разницу

OAuth 2.0 — это про ключ от квартиры. OpenID Connect — это про паспорт + ключ. OAuth говорит «у тебя есть право войти», OIDC ещё и подтверждает «и вот кто ты такой». Практически все кнопки «Войти через X» используют OIDC.

Redis vs Memcached — когда что выбирать

Оба — in-memory хранилища. БД отдаёт данные за 50–200 мс, кэш — за 0.1–2 мс. Разница между Redis и Memcached принципиальная.

Redis
Многофункциональный — выбирают в 95% случаев
  • Поддерживает структуры: String, List, Hash, Set, Sorted Set, Stream
  • Персистентность (RDB снапшоты + AOF журнал) — данные не теряются при рестарте
  • Pub/Sub, очереди задач, распределённые блокировки
  • TTL на любой ключ, Lua-скрипты, транзакции
  • Используют: сессии, кэш, очереди, rate limiting, leaderboards
Memcached
Простой — только кэш и ничего лишнего
  • Только строки — никаких сложных структур
  • Нет персистентности — при рестарте всё теряется
  • Многопоточный — чуть быстрее на простых операциях
  • Меньше памяти на overhead
  • Выбирают: когда нужен только простой key-value кэш с максимальной скоростью

Стратегии кэширования

Выбор стратегии — архитектурное решение, которое СА закладывает в требования. От неё зависит консистентность данных и нагрузка на БД.

Cache-Aside (Ленивое)
Приложение управляет кэшем
1. Запрос → смотрим в кэш
2. Cache miss → идём в БД
3. Пишем результат в кэш
4. Следующий запрос — cache hit
✅ Данные в кэше только то, что реально запрашивают. Кэш не падает → просто идём в БД
Использовать: каталог товаров, профили пользователей
Write-Through (Сквозная запись)
Запись всегда через кэш
При записи → одновременно пишем в кэш И в БД. Чтение всегда из кэша — там актуальные данные.
✅ Кэш всегда консистентен с БД
❌ Запись медленнее — два хранилища синхронно
Использовать: корзина покупок, настройки пользователя
Write-Behind (Отложенная запись)
Сначала кэш, потом БД асинхронно
При записи → только в кэш (мгновенно). Фоновый процесс батчами синхронизирует изменения с БД.
✅ Максимальная скорость записи
❌ Риск потери данных если кэш упадёт до синхронизации
Использовать: счётчики, аналитика, аудит-логи
Cache Invalidation — самая сложная задача

«В CS есть только две сложные проблемы: именование переменных и инвалидация кэша» — Фил Карлтон. Когда данные в БД изменились, кэш должен узнать об этом. Стратегии: TTL (кэш устаревает через N секунд сам), Event-driven (при изменении в БД публикуем событие, сервис кэша слушает и сбрасывает ключи), Cache-busting (при деплое меняем версию ключей). NFR: «Данные каталога обновляются в кэше в течение 5 минут после изменения в БД».

Что НЕЛЬЗЯ писать в логи (PII / Secrets)

Пароли, токены, номера карт, персональные данные (ФИО, email, номер телефона) — никогда в логах. Это нарушение 152-ФЗ и GDPR. NFR на логирование всегда должен включать: «Логи не содержат персональные данные пользователей и секретные ключи».

ELK Stack и стек наблюдаемости

В микросервисах логи раскиданы по десяткам серверов. Три столпа наблюдаемости (observability): Logs (что случилось), Metrics (как работает система), Traces (где застрял запрос).

📋
Logs → ELK / Loki
Logstash/Fluentd собирают, Elasticsearch индексирует, Kibana визуализирует. Grafana Loki — лёгкая альтернатива. Ищи по Trace-ID чтобы проследить запрос через все сервисы.
📊
Metrics → Prometheus + Grafana
Prometheus собирает числовые метрики каждые N секунд (CPU, память, RPS, error rate, latency). Grafana строит дашборды. Алерты срабатывают когда метрика выходит за порог.
🔍
Traces → Jaeger / Zipkin
Distributed Tracing — полный путь запроса через микросервисы с таймингами каждого шага. API Gateway генерирует Trace-ID, все сервисы передают его дальше в заголовках.
🚨
Alerting → PagerDuty / Alertmanager
Правила алертов: error rate > 1% → предупреждение, error rate > 5% → разбудить дежурного. SLA: 99.9% uptime = не более 8.7 часов даунтайма в год.

Бонус · Глоссарий
Глоссарий СА — 40 терминов
Термины, которые должен знать каждый системный аналитик.
API (Application Programming Interface)
▼
Контракт взаимодействия между системами. Набор правил: как вызвать функцию, что передать и что получить в ответ.
Артефакт анализа
▼
Любой документ, схема или прототип, созданный аналитиком: ТЗ, Use Case, ERD, Sequence Diagram, макет.
Backlog (Бэклог)
▼
Упорядоченный список всего, что нужно сделать. Product Backlog — весь список продукта. Sprint Backlog — то, что берём в текущий спринт.
BFF (Backend For Frontend)
▼
Паттерн: отдельный API Gateway для каждого типа клиента (Web, Mobile, TV), возвращающий только нужные данные.
Бизнес-правило
▼
Политика или ограничение, независимое от наличия ИТ-системы. Например: «скидка VIP не суммируется с промокодом».
CAP-теорема
▼
Распределённая система может гарантировать не более двух из трёх: Consistency (согласованность), Availability (доступность), Partition tolerance (устойчивость к разрывам).
Circuit Breaker
▼
Паттерн «предохранитель». Если сторонний сервис недоступен — автоматически прекращаем к нему обращаться на N секунд, чтобы не положить всю систему в ожидании таймаута.
CRUD
▼
Create, Read, Update, Delete — четыре базовые операции с данными. Большинство API реализуют именно их.
Декомпозиция
▼
Разбивка большой задачи (Эпик) на маленькие (User Story). Задача считается готовой к разработке, когда её можно выполнить за один спринт.
Domain (Домен)
▼
Бизнес-область со своей логикой. «Заказы», «Оплата», «Уведомления» — разные домены. Каждый микросервис желательно ограничивать одним доменом (DDD — Domain Driven Design).
Endpoint (Эндпоинт)
▼
Конкретный URL + HTTP-метод, принимающий запросы. Например: POST /api/v1/orders — это один эндпоинт.
ERD (Entity Relationship Diagram)
▼
Схема базы данных: таблицы (сущности), их поля и связи между ними (1:1, 1:N, M:N). Обязательный артефакт при проектировании БД.
Eventual Consistency
▼
«Конечная согласованность» — данные в распределённой системе сойдутся к единому значению не мгновенно, а через некоторое время. Приемлемо для лайков, но не для балансов.
Идемпотентность
▼
Свойство операции: многократный вызов даёт тот же результат, что и однократный. GET, PUT, DELETE — идемпотентны. POST — нет.
JSON (JavaScript Object Notation)
▼
Текстовый формат обмена данными. Читаем для человека, прост для машины. Стандарт де-факто для REST API.
JWT (JSON Web Token)
▼
Компактный токен авторизации из трёх частей: Header.Payload.Signature. Сервер верифицирует подпись без обращения к БД при каждом запросе.
Kafka / RabbitMQ
▼
Брокеры сообщений. Kafka — журнал событий (долго хранит, много читателей). RabbitMQ — умная маршрутизация задач между воркерами.
Loose Coupling (слабая связанность)
▼
Принцип: компоненты системы должны зависеть друг от друга как можно меньше. Изменение одного не ломает другой. Главная цель микросервисной архитектуры.
MVP (Minimum Viable Product)
▼
Минимально жизнеспособный продукт — наименьший набор функций, которые несут реальную ценность для пользователя и позволяют проверить гипотезу.
NFR (Non-Functional Requirements)
▼
Нефункциональные требования — атрибуты качества системы: производительность, надёжность, безопасность, масштабируемость, совместимость.
OpenAPI / Swagger
▼
Стандарт описания REST API в формате YAML/JSON. Swagger UI генерирует по этому файлу интерактивную документацию, где можно сразу тестировать эндпоинты.
Polling vs Webhook
▼
Polling — клиент периодически опрашивает сервер «что нового?» (неэффективно). Webhook — сервер сам уведомляет клиента POST-запросом, когда событие произошло.
RFC / ADR
▼
RFC (Request for Comments) — документ с предложением изменения для обсуждения. ADR (Architecture Decision Record) — фиксация принятого архитектурного решения с обоснованием.
SLA / SLO / SLI
▼
SLA — юридический договор (штрафы за нарушение). SLO — внутренняя цель команды (99.9% uptime). SLI — реальная измеренная метрика (сейчас 99.95%). SLI должен достигать SLO.
Stale Data (устаревшие данные)
▼
Данные, которые устарели с момента последнего обновления кэша или реплики. Важно предупреждать пользователя (пометка «обновлено 5 мин назад»).
Stateless / Stateful
▼
Stateless — сервер не хранит состояние клиента между запросами (REST). Каждый запрос самодостаточен. Stateful — сервер помнит контекст (WebSocket, SSH-сессии).
Strangler Fig Pattern
▼
Паттерн постепенной замены монолита. Новый микросервис разворачивается рядом, часть трафика переводится через API Gateway. Монолит «усыхает» постепенно.
Throughput / Latency
▼
Throughput — сколько запросов система обрабатывает в единицу времени (RPS). Latency — время ответа на один запрос (мс). Оба NFR важны: высокий throughput при высокой latency — плохо.
UAT (User Acceptance Testing)
▼
Приёмочное тестирование. Стейкхолдеры или конечные пользователи проверяют систему на соответствие бизнес-требованиям. СА участвует как фасилитатор.
Velocity (скорость команды)
▼
Среднее количество Story Points, которое команда закрывает за спринт. Используется для планирования: сколько задач войдёт в следующий спринт.

Архитектура и Интеграции

1. Что такое Идемпотентность? Зачем она нужна?
▼

Свойство операции давать один и тот же результат при многократном вызове. Например, GET и PUT идемпотентны, а POST — нет. Нужна для защиты от двойных списаний (если у клиента дрогнул интернет и он дважды нажал "Оплатить", спасет Idempotency-Key).

2. В чем разница между Синхронным и Асинхронным взаимодействием?
▼

Синхронно (REST/gRPC) — сервис отправил запрос и заблокировался, ожидая ответа (как звонок по телефону). Асинхронно (Kafka/RabbitMQ) — сервис положил сообщение в очередь и сразу пошел делать другие дела, не дожидаясь ответа (как отправка Email).

3. SOAP vs REST? В чем ключевая разница?
▼

SOAP — это строгий протокол, использует только XML, имеет встроенную безопасность (WS-Security) и жесткие контракты WSDL. REST — это архитектурный стиль, работает поверх HTTP, использует любые форматы (чаще JSON), легковесный и stateless.

4. Что такое API Gateway?
▼

Единая точка входа для клиентов в микросервисной архитектуре. Выполняет маршрутизацию запросов, проверку JWT токенов (Auth), ограничение нагрузки (Rate Limiting) и логирование.

5. Что такое Webhooks?
▼

Это "обратный вызов" (Reverse API). Вместо того чтобы клиент каждую секунду спрашивал сервер "Статус изменился?" (Polling), клиент дает серверу свой URL. И когда статус реально изменится, сервер сам сделает POST-запрос на этот URL.

6. Что такое CAP-теорема?
▼

В распределенной системе можно гарантировать максимум два из трех свойств: Consistency (Согласованность данных), Availability (Доступность системы) и Partition tolerance (Устойчивость к разрывам сети). Так как сеть всегда может порваться, системы делятся на CP (банки, отказ вместо неверных данных) и AP (соцсети, отдаем старые данные, но работаем).

Микросервисы и Брокеры

7. Kafka или RabbitMQ? Что выбрать?
▼

Kafka (Журнал событий) — если нужно хранить историю, обрабатывать стриминг Big Data и иметь множество независимых читателей одного события. RabbitMQ (Умная маршрутизация) — для распределения точечных задач (например, отправить письмо), где сообщение удаляется сразу после прочтения воркером.

8. Что такое Dead Letter Queue (DLQ)?
▼

Очередь мертвых писем. Если наш сервис (Consumer) не может обработать сообщение из-за ошибки в коде или невалидного JSON, после N попыток он перекладывает это сообщение в DLQ, чтобы не застопорить всю остальную очередь нормальных сообщений.

9. Как работает паттерн Saga?
▼

Заменяет классические ACID транзакции в микросервисах. Состоит из цепочки локальных транзакций. Если одна из них падает (например, нет билетов на самолет), система запускает "компенсирующие транзакции" в обратном порядке (вернуть деньги за отель, вернуть деньги на карту).

10. Что такое Circuit Breaker?
▼

Паттерн "Предохранитель". Защищает систему от каскадных сбоев. Если внешний сервис "лежит" и запросы к нему падают по таймауту, Circuit Breaker переходит в состояние Open и сразу возвращает ошибку, давая внешнему сервису время на восстановление.

Базы Данных

11. ACID vs BASE?
▼

ACID (Atomicity, Consistency, Isolation, Durability) — строгие гарантии реляционных БД (SQL). BASE (Basically Available, Soft state, Eventual consistency) — модель NoSQL баз данных, где ради скорости и масштабируемости жертвуют строгой консистентностью (данные станут консистентны, но "чуть позже").

12. Как работают B-Tree индексы?
▼

B-Tree (сбалансированное дерево) — сортирует данные для быстрого поиска за логарифмическое время O(log N). Идеален для поиска по диапазону (>, <, BETWEEN). Не работает, если искать подстроку в начале или середине слова (LIKE '%xyz').

13. Уровни изоляции транзакций в БД?
▼

1. Read Uncommitted (грязное чтение разрешено).
2. Read Committed (видим только закоммиченные данные).
3. Repeatable Read (защита от изменения строки другой транзакцией во время нашего чтения).
4. Serializable (строгая блокировка таблиц, транзакции идут друг за другом).

14. Что такое Нормализация базы данных?
▼

Процесс организации данных для уменьшения дублирования. 1NF: атомарные ячейки (без массивов в ячейке). 2NF: каждая колонка зависит от всего первичного ключа. 3NF: нет транзитивных зависимостей (колонки не вычисляются друг из друга, например, сумма=цена*кол-во).

Безопасность и Сеть

15. Аутентификация (AuthN) vs Авторизация (AuthZ)?
▼

Аутентификация проверяет КТО вы (логин/пароль). Авторизация проверяет ЧТО ВАМ МОЖНО делать (права доступа: Админ или Юзер).

16. Из чего состоит JWT (JSON Web Token)?
▼

Из трех частей, разделенных точками: Header (тип и алгоритм шифрования), Payload (сами данные: user_id, роли, срок годности в Base64 — их видят все!) и Signature (хэш первых двух частей с секретным ключом сервера для защиты от подделки).

17. Коды ответов HTTP (Краткая шпаргалка)?
▼

200 OK, 201 Created.
400 Bad Request (ошибка валидации), 401 Unauthorized (нет токена), 403 Forbidden (нет прав), 404 Not Found.
500 Internal Server Error (код упал), 502 Bad Gateway (упал нижестоящий сервис), 504 Gateway Timeout.

18. Симметричное vs Асимметричное шифрование?
▼

Симметричное: один и тот же ключ используется и для шифрования, и для расшифровки. Асимметричное (RSA): есть Публичный ключ (раздаем всем, им шифруют) и Приватный ключ (лежит в сейфе, только им можно расшифровать). На этом построен HTTPS.

19. Что такое CORS?
▼

Механизм безопасности браузера. Запрещает скриптам с домена A.com делать запросы к API на домене B.com, если сервер B.com явно не пришлет заголовок Access-Control-Allow-Origin.

Процессы и Метрики

20. User Story vs Use Case?
▼

User Story (Agile): Короткое описание фичи с позиции ценности для пользователя ("Как клиент, хочу..."). Use Case (Enterprise): Детальный пошаговый сценарий взаимодействия системы и актора с описанием всех альтернативных веток и ошибок.

21. Бизнес-требования vs Системные требования?
▼

Бизнес: "Зачем мы это делаем?" (цели, метрики, деньги). Системные: требования к железу, ОС, СУБД, сетям, на которых будет работать готовое ПО.

22. Что такое SLA, SLO и SLI?
▼

SLA (Agreement) — юридический договор с клиентом (штрафы, если система лежит). SLO (Objective) — внутренняя цель команды (например, 99.9% аптайм). SLI (Indicator) — реальная текущая метрика в мониторинге (например, сейчас аптайм 99.95%).

23. Что такое RTO и RPO?
▼

Метрики восстановления после аварии. RTO (Recovery Time Objective): за сколько часов мы обязаны поднять сервера. RPO (Recovery Point Objective): за сколько часов/минут мы готовы потерять данные (насколько часто делаем бэкапы).

24. Как разрешать конфликты стейкхолдеров?
▼

1. Провести совместный Воркшоп. 2. Оцифровать последствия каждого решения (сроки/деньги). 3. Если компромисса нет — эскалировать на Спонсора проекта (Product Owner), так как только он владеет бюджетом.

25. Метод STAR на собеседовании?
▼

Отвечай на поведенческие вопросы по структуре: Situation (какая была проблема в проекте) -> Task (какую задачу поставили мне) -> Action (мои конкретные действия: схемы, маппинг, созвоны) -> Result (цифры, метрики успеха, задеплоенный релиз).

❌ Ошибка 4

✅ Правильно

❌ Ошибка 5

✅ Правильно

❌ Ошибка 6

❌ Ошибка 7
Описал что система должна делать, но не описал что NOT должна делать. Разработчики добавили «полезные» фичи.

В разделе Ограничения явно пиши: «Система НЕ должна...» — это экономит время и деньги.
❌ Ошибка 8
Показал диаграмму заказчику. Он не понял ничего и потерял доверие к аналитику.

📋 Бизнес-контекст

  • Описана бизнес-проблема, которую решает фича
    Зачем это вообще нужно
  • Указана измеримая бизнес-цель (KPI, метрика успеха)
    Как поймём что фича удалась
  • Все стейкхолдеры выявлены и опрошены
    Никто не скажет «меня не спросили» на демо
  • Есть глоссарий — ключевые термины расшифрованы
    «Клиент» и «Пользователь» — это разные сущности?
  • Scope чётко обозначен: что входит и что НЕ входит в фичу
    Явный Out of Scope предотвращает scope creep
  • Риски и допущения задокументированы
    Что может пойти не так и от чего зависим

📐 Функциональные требования

  • Все User Story написаны по формату «Как X, хочу Y, чтобы Z»
    Роль, действие, ценность — все три элемента
  • У каждой User Story есть критерии приёмки (Given/When/Then)
    Минимум: 1 позитивный + 1 негативный сценарий
  • Описаны все Alternative Flows (что если что-то пошло не так)
    Таймаут, недоступность сервиса, невалидные данные
  • Явно указано, что система NOT должна делать
    Ограничения области видимости фичи
  • Бизнес-правила вынесены отдельным списком
    Формулы, лимиты, условия, права доступа по ролям

⚡ Нефункциональные требования

  • Указана нагрузка: RPS, количество пользователей
    Не «много» — а конкретные цифры
  • Указано время отклика API (p95, p99)
    Например: ≤ 300ms для 95% запросов
  • Описаны требования к доступности (SLA Uptime)
    99.9% = 43 минуты простоя в месяц
  • Описана безопасность: кто видит данные, шифрование
    GDPR, 152-ФЗ, роли доступа к данным

🔗 Интеграции и данные

  • Все внешние системы и API перечислены
    Кто вызывает кого и с какими данными
  • Есть Sequence Diagram для каждого интеграционного сценария
    Синхрон vs асинхрон, обработка ошибок
  • Описана модель данных (ERD или таблица полей)
    Какие таблицы создаём/меняем, типы данных
  • Есть маппинг данных: откуда берётся каждое поле
    Источник → поле в API → поле в БД
  • Контракт API описан (или ссылка на Swagger)
    Методы, параметры, коды ответов, примеры

✅ Качество документа

  • Нет расплывчатых слов: «быстро», «удобно», «много»
    Всё заменено конкретными цифрами
  • Разные люди читают требования одинаково
    Провёл review с коллегой или разработчиком
  • Требования не противоречат друг другу
    Прошёлся по списку на логические конфликты
  • Есть раздел с открытыми вопросами (Open Issues)
    Что ещё не прояснено и кто отвечает
  • Документ прошёл согласование у PO
    Подпись/аппрув в Jira или Confluence
  • QA ознакомлен и может написать тест-кейсы по ТЗ
    Если QA не понимает — ТЗ написано плохо
  • UI/UX макеты прикреплены к требованиям (ссылки на Figma)
    Разработчик должен видеть экран рядом с требованием
  • RTM обновлена — новые требования связаны с тест-кейсами
    Нет трассировки = нет гарантии что всё проверено
  • Версия документа присвоена и статус актуален
    v1.0 Draft / v1.1 In Review / v1.2 Approved
